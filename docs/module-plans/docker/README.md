## 1. 目标定位

Docker 模块的目标不是只做一个容器列表页，也不是简单复刻 Docker Desktop，而是做成“**AI 原生工程工作台里的 Docker 连接级工作区**”。

它要服务 OmniPanel 的一句话定位：

> 一个窗口，管理服务器、数据库、容器与工作流；一个 AI，贯穿开发运维上下文。

因此 Docker 模块必须同时满足四个产品要求：

- **上下文不断掉**：容器、宿主机、SSH、日志、文件、命令和 AI 分析应在同一任务上下文中连续发生。
- **小而优**：优先打通高频闭环，不堆低频高级能力；默认界面轻、反馈明确、学习成本低。
- **专业可信**：生产环境操作、删除、清理、重启等高风险动作必须确认、审计、可回看。
- **本地优先**：连接、历史、审计和凭据优先保存在本机，凭据走系统 Keychain，不依赖云端服务。

对于任意一个 Docker 连接，无论它来自本地 Engine、远程 Engine、SSH 宿主机，还是 1Panel / 宝塔 / Portainer 适配，用户都应该看到一致的资源工作区结构：

```text
Docker
└── [连接 A | 连接 B | 连接 C]
    ├── 总览
    ├── Compose
    ├── 容器
    ├── 镜像
    ├── 卷
    └── 网络
```

这意味着 Docker 模块的主对象不再是“容器”，而是“连接”和“资源域”。容器、Compose、镜像、卷、网络都只是某个 Docker 连接下的一级资源视图。

首个可用版本的核心体验应当是：用户选中一个 Docker 连接后，可以立即看到真实容器状态，进入日志，打开容器终端，执行启动/停止/重启，并能把异常上下文交给 AI 或跳回 SSH / Server 模块继续排障。

## 2. 当前现状

### 2.1 已有基础

- 前端已有 Docker 模块入口与连接切换基础
- 当前页面已有容器列表、筛选、详情抽屉、基础动作按钮原型
- 统一连接模型、动作执行引擎、SSH 与终端能力已经存在，可作为 Docker 模块落地基础设施

### 2.2 当前问题

- 当前 Docker 页面主体仍是 mock 容器列表
- Compose、镜像、卷、网络尚未升级为独立资源域
- Docker 动作更接近“shell 执行命令”，不是“统一 Docker 能力服务”
- 多来源接入还没有抽象成统一 adapter
- 缺少连接能力探测，前端无法根据来源能力进行按钮显隐或降级

### 2.3 当前实现判断

当前 [DockerPanel.tsx](../../../frontend/src/modules/docker/DockerPanel.tsx) 更适合作为后续 `DockerContainersView` 的原型基础，而不应继续承担整个 Docker 模块入口的职责。

## 3. 核心决策

### 3.1 Docker 模块按连接工作区组织

- Docker 首页不是“全局容器池”
- Docker 首页应先选择连接，再进入连接内工作区
- 每个连接都拥有独立的资源页签、缓存、详情上下文和动作范围

### 3.2 Compose 必须是一级资源域

- Compose 不应只是页面顶部一个按钮
- Compose 是用户理解“应用级编排”的主入口
- 后续的聚合日志、项目级动作、服务依赖关系都依赖这一层结构
- 但首个版本不急于做完整 YAML 编辑器，先完成项目识别、服务列表、项目级 up/down/restart 与聚合日志入口

### 3.3 连接来源差异由后端屏蔽

前端统一只看“Docker 连接”和“Docker 资源”，不直接理解底层是：

- 本地 Engine
- 远程 Engine API
- SSH 宿主机
- 面板 API 适配

来源差异全部放在后端 adapter 内部处理。

### 3.4 第一阶段优先做 `local-engine` 与 `ssh-engine`

原因：

- 当前代码基础最容易复用
- 最接近 OmniPanel 的真实用户场景
- 能最快把容器列表、日志、exec、Compose 基础能力打通

### 3.5 先做能力闭环，再做技术升级

第一阶段优先保证：

- 列表能拉到真实数据
- 日志能看
- 容器能 exec
- 关键动作可执行

技术路径按来源分层：

- `local-engine`：优先使用 Rust `bollard` 直连 Docker Engine，符合 PRD 技术栈。
- `ssh-engine`：优先复用现有 SSH 能力调用远程 `docker` / `docker compose` CLI，快速覆盖真实远程宿主机场景。
- `remote-engine`：作为后续增强，通过 Docker TCP/TLS API 接入。
- `panel-adapter`：放到面板集成阶段，通过 1Panel / 宝塔 / Portainer adapter 映射统一资源模型。

不引入 `dockerode`、Node WebSocket 服务、Redis、Prometheus 等与当前 Rust/Tauri 桌面架构不匹配的依赖，除非未来架构明确需要。

### 3.6 UI 原型先收敛为真实工作流

当前 `DockerPanel.tsx` 的 mock 容器列表适合作为视觉原型，但第一阶段不继续扩大 mock。后续改造顺序为：

1. 先拆出连接选择、工作区页签、容器列表、详情抽屉等组件。
2. 再接入 Tauri IPC 的真实数据，并保留空态 / 错误态 / 降级态。
3. 最后补 Compose、镜像、卷、网络等资源域。

## 4. 模块边界

### 4.1 本模块负责

- Docker 连接级工作区
- Compose / 容器 / 镜像 / 卷 / 网络 管理
- 容器日志、容器终端、容器基础操作
- 连接能力探测与资源视图降级
- 与 SSH / Server / AI 的跳转联动

### 4.2 本模块不直接负责

- 完整服务器监控面板
- 通用面板连接管理界面
- 云镜像仓库高级治理
- Kubernetes 编排能力
- 生产环境审批流本身的实现细节

这些能力可与 Docker 模块联动，但不应耦合在 Docker 工作区内部实现。

## 5. 模块结构总览

### 5.1 连接内一级页签

- **总览**：连接状态、资源总数、异常资源、最近事件、快捷动作
- **Compose**：项目列表、项目详情、聚合日志、项目级动作
- **容器**：容器列表、容器详情、日志、终端、生命周期管理
- **镜像**：镜像列表、拉取、删除、清理、推送
- **卷**：卷列表、卷详情、清理、关联容器
- **网络**：网络列表、网络详情、删除、关联容器

### 5.2 后续可扩展资源域

以下能力先不做一级页签，但架构上应预留扩展空间：

- 事件流
- 文件系统浏览
- 镜像构建
- Registry 凭据
- Docker 设置 / 连接诊断

## 6. 与其他模块的关系

### 6.1 与 SSH 模块

- `ssh-engine` 来源依赖 SSH 连接作为能力载体
- Docker 连接可绑定 SSH 连接
- 从 Docker 工作区可跳转 SSH 终端或远程文件

### 6.2 与 Server 模块

- 总览中可引用服务器资源统计与异常信息
- Docker 连接可在服务器页签中作为联动入口
- 容器异常排障可跳回服务器监控与日志

### 6.3 与 AI 模块

- 容器日志、状态、错误码、Compose 项目可作为 AI 上下文
- 后续支持“发送给 AI 排障”“生成修复命令草稿”
- AI 生成的 Docker 命令仍走统一动作确认与审计

### 6.4 与动作执行 / 审计体系

- 高风险动作需要统一确认
- 容器删除、镜像 prune、卷清理、网络删除等动作需要纳入审计
- 前后端不能绕过统一的风险判断策略

## 7. 目标交付形态

完整交付后，Docker 模块应具备以下产品特征：

- 从连接视角管理 Docker，而不是从单个资源视角零散堆按钮
- 既能满足本地开发机调试，也能满足远程宿主机运维
- 既能看单个容器，也能看 Compose 项目整体
- 能以统一结构承载不同来源的 Docker 能力
- 后续能无缝接入 1Panel / 宝塔 / Portainer 等适配层

## 8. 建议开发顺序

建议按以下顺序推进：

1. 先完成连接级工作区结构与前端页签骨架
2. 优先打通 `local-engine` / `ssh-engine` 的真实容器链路
3. 再补 Compose 项目视图与镜像视图
4. 最后补卷、网络、事件流与更深的联动能力

详细拆解见 [迭代计划](./milestones.md)。

## 9. 成果定义

本目录内文档完成后，Docker 模块设计应达到以下状态：

- 连接模型明确
- 资源模型明确
- 页面层级明确
- 组件拆分明确
- IPC / 命令接口边界明确
- 迭代顺序与验收标准明确

如果后续开发仍出现“大方向不清楚”，优先回到本目录修订文档，而不是继续在聊天里累积口头结论。
