## 1. 目标定位

Server 模块的目标是成为"服务器统一工作区"，无论服务器是通过 SSH 直连、1Panel、还是宝塔面板接入，用户都应该看到一致的管理视图：

```text
Server
└── [服务器 A | 服务器 B | 服务器 C]
    ├── 监控
    ├── 进程
    ├── 服务
    ├── 日志
    ├── 文件
    └── (面板资源: 网站/数据库/证书/任务)
```

Server 模块与 Docker 模块共享"连接级工作区"的设计模式，但 Server 模块更侧重于系统层面的资源管理和监控。

## 2. 当前现状

### 2.1 已有基础

- 已有 ServerPanel.tsx 实现了基础框架
- 已有 ServerSidebar.tsx 组件用于服务器选择
- 已有 monitor、processes、services、logs 四个页签
- 与 actionStore 集成，支持动作执行和审计
- 使用 mock 数据展示 UI 原型

### 2.2 当前问题

- 当前数据均为 mock，非真实服务器数据
- SSH 直连采集能力尚未实现
- 面板集成（1Panel/宝塔）尚未实现
- 历史数据存储尚未实现
- 告警阈值管理尚未实现
- 批量操作尚未实现
- 进程/服务管理仅有查看，缺少操作能力

### 2.3 当前实现判断

当前 [ServerPanel.tsx](../../../frontend/src/modules/server/ServerPanel.tsx) 已完成 UI 原型设计，适合作为后续真实数据接入的基础框架。需要重构数据层，从 mock 数据切换为真实 API 调用。

## 3. 核心决策

### 3.1 服务器按连接工作区组织

- Server 首页不是"全局服务器池"
- Server 首页应先选择服务器/面板，再进入服务器内工作区
- 每个服务器都拥有独立的监控数据、进程列表、服务状态、日志上下文

### 3.2 数据采集双轨制

Server 模块支持两种数据采集方式：

- **SSH 直连采集**：通过 SSH 连接直接执行系统命令采集数据（top、df、journalctl 等）
- **面板 API 采集**：通过 1Panel/宝塔等面板的 API 获取监控和管理数据

两种方式采集的数据最终映射到同一套资源模型。

### 3.3 面板集成优先级

| 面板 | 优先级 | 接入能力 |
|------|--------|----------|
| 1Panel | P1 | 服务器状态、容器、网站、证书、计划任务、应用商店 |
| 宝塔面板 | P1 | 服务器状态、网站、数据库、FTP、计划任务、防火墙、SSL |

### 3.4 第一阶段优先 SSH 直连

原因：

- SSH 直连是 OmniPanel 的核心能力，最容易复用现有基础设施
- SSH 直连可覆盖任何 Linux 服务器，不依赖特定面板
- 能最快把监控、进程、服务、日志能力打通

### 3.5 面板集成为第二阶段

在 SSH 直连能力稳定后，再接入 1Panel 和宝塔面板，利用其 API 获取更丰富的管理能力。

## 4. 模块边界

### 4.1 本模块负责

- 服务器级工作区
- 系统监控（CPU/内存/磁盘/网络）
- 进程管理（查看、搜索、终止）
- 服务管理（systemd 服务启停、状态查看）
- 日志查看（journalctl、tail -f）
- 面板资源同步（网站、数据库、证书、计划任务）
- 与 SSH 模块的跳转联动
- 与 Docker 模块的数据联动

### 4.2 本模块不直接负责

- SSH 连接管理界面（由 SSH 模块负责）
- 容器内部管理（由 Docker 模块负责）
- 完整的网站/数据库管理（由面板负责，本模块只做同步和展示）
- 服务器硬件监控（IPMI/ILO/iDRAC）
- 云服务商原生监控集成

这些能力可与 Server 模块联动，但不应耦合在 Server 工作区内部实现。

## 5. 模块结构总览

### 5.1 服务器内一级页签

- **监控**：CPU/内存/磁盘/网络实时图表、历史数据、告警状态
- **进程**：进程列表、搜索过滤、CPU/内存排序、进程终止
- **服务**：systemd 服务列表、服务状态、服务启停重启
- **日志**：journalctl 日志、实时 tail、搜索过滤、日志导出
- **文件**：SFTP 文件浏览器（跳转 SSH 模块）
- **面板资源**（如已接入面板）：网站、数据库、证书、计划任务

### 5.2 后续可扩展功能域

以下能力先不做一级页签，但架构上应预留扩展空间：

- 定时任务管理（crontab）
- 防火墙规则管理（iptables/ufw）
- 用户管理
- 磁盘分析
- 证书管理
- 批量操作台

## 6. 与其他模块的关系

### 6.1 与 SSH 模块

- SSH 直连是 Server 模块的核心数据采集方式
- 服务器可绑定 SSH 连接
- 从 Server 工作区可跳转 SSH 终端或远程文件
- SFTP 文件管理复用 SSH 模块能力

### 6.2 与 Docker 模块

- Server 总览中可引用 Docker 容器统计
- 容器异常可跳转 Server 监控排障
- Docker 连接可与 Server 关联
- 同一台服务器的 Docker 和 Server 数据可联动展示

### 6.3 与 AI 模块

- 服务器日志、监控指标、错误码可作为 AI 上下文
- 后续支持"发送给 AI 分析""生成修复命令草稿"
- AI 生成的系统命令仍走统一动作确认与审计

### 6.4 与面板模块（1Panel/宝塔）

- 面板资源同步到 Server 工作区统一展示
- 面板操作（如重启服务）通过面板 API 执行
- 面板与 SSH 连接可绑定，实现能力互补

### 6.5 与动作执行 / 审计体系

- 高风险操作（进程终止、服务重启、防火墙修改）需要统一确认
- 所有操作纳入审计日志
- 前后端不能绕过统一的风险判断策略

## 7. 目标交付形态

完整交付后，Server 模块应具备以下产品特征：

- 从服务器视角管理资源，而不是从单个功能视角零散堆砌
- 既能满足 SSH 直连轻量场景，也能满足面板深度管理场景
- 既能看实时监控，也能追溯历史数据
- 能以统一结构承载不同接入方式的服务器
- 后续能无缝接入更多面板适配层

## 8. 建议开发顺序

建议按以下顺序推进：

1. 先完成 SSH 直连采集的真实数据链路
2. 再完善监控面板和历史数据存储
3. 再补进程管理、服务管理的完整操作能力
4. 再补日志查看和导出能力
5. 最后接入 1Panel/宝塔面板

详细拆解见 [迭代计划](./milestones.md)。
