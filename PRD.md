## 1. 产品概述

### 1.1 产品愿景

打造一个**AI 原生的个人工程工作台**，成为开发者连接本地环境、远程服务器、数据库、容器、协议调试、工程知识和团队工作流的统一入口。

OmniPanel 的目标不是做一个“运维工具集合”，而是重新定义 coder 的日常工程入口。过去，开发者在终端、SSH、数据库客户端、Docker 面板、HTTP 调试器、串口工具、监控系统和 AI 对话框之间来回切换；未来，这些工作应该围绕同一个任务上下文，在一个轻量、可信、优雅的工程工作台中连续发生。

我们希望 OmniPanel 成为 coder 电脑里的**个人工程操作系统**：它理解环境，连接资源，承载操作，记录过程，辅助决策，沉淀知识，并通过 AI 把人、工具、系统和经验组织到一起。

我们的产品思想是：**All in One，小而全而优而美**。

- **All in One**：一个应用承载个人开发者和小团队日常运维所需的完整能力。
- **小**：安装轻、启动快、界面克制、学习成本低，不把复杂度直接甩给用户。
- **全**：能力覆盖日常开发运维的完整闭环，但每个能力都围绕真实工作流精心取舍。
- **优**：优先把高频场景做到稳定、顺滑、可信，不追求低价值功能堆砌。
- **美**：高信息密度不等于粗糙，专业工具也要有清晰层级、顺滑交互、统一审美和可长期使用的舒适感。

最终形态应当是：用户打开 OmniPanel，就像打开自己的工程驾驶舱。所有环境、连接、资源、命令、日志、数据和知识都在一个地方，AI 作为随行专家理解当前上下文，帮助用户更快、更安全、更优雅地完成工作。

我们的长期愿景是：**让未来每一位 coder 的电脑上，都安装 OmniPanel。**

它不只是一个运维工具，而是开发者进入本地、远程、数据、容器、协议、AI 和团队工作流的统一入口。只要一个人写代码、调服务、查数据、连服务器、看日志、部署应用，OmniPanel 就应该成为他电脑里的默认工程工作台。

### 1.1.1 产品使命

> 让每一位 coder 用一个优雅、可信、AI 原生的工作台，完成从开发到交付、从调试到复盘的完整工程闭环。

OmniPanel 要解决的不是“少打开几个软件”这么简单的问题，而是开发者工程效率、上下文连续性、生产安全、经验沉淀和团队协作的系统性问题。

### 1.1.2 产品层次

| 层次 | 定位 | 价值 |
|------|------|------|
| 工具层 | 终端、SSH、数据库、Docker、HTTP、串口、MQTT、WebSocket | 替代分散工具，减少切换 |
| 工作流层 | 工作区、快捷脚本、数据工作流、操作草稿箱、任务中心 | 将零散操作组织成连续流程 |
| 智能层 | AI API、本地模型、CLI Agent、上下文快照、操作链 | 让 AI 理解现场并参与决策 |
| 知识层 | 时间线、复盘、命令片段、故障案例、团队共享 | 将一次性经验沉淀为可复用资产 |
| 协同层 | Hub、Web、手机端、审批、审计、执行代理 | 从个人效率延展到团队和企业工程入口 |

### 1.2 产品名称

**OmniPanel**（暂定）

- Omni = 全部、全能
- Panel = 面板、工作台

### 1.3 一句话描述

> 一个窗口，管理服务器、数据库、容器与工作流；一个 AI，贯穿开发运维上下文。

### 1.4 产品原则

| 原则 | 说明 |
|------|------|
| 一个工作台 | 所有核心能力必须服务于统一工作台，而不是互相割裂的子应用 |
| 上下文贯通 | 终端命令、SSH 主机、数据库连接、容器、日志、监控和 AI 对话共享上下文 |
| 本地优先 | 凭据、历史、配置和敏感数据优先保存在本机，可选同步但不强制上云 |
| 专业可信 | 危险操作明确提示，AI 只建议不越权，关键命令必须可审计、可回滚、可复盘 |
| 高密度且美观 | 保留专业工具的信息密度，同时通过布局、动效、字体、色彩和状态反馈提升长期使用体验 |
| 渐进复杂 | 新手可以快速连接一台服务器，专家可以组织复杂环境、批量操作和自动化工作流 |

### 1.5 目标用户

| 用户角色 | 使用场景 | 核心痛点 |
|---------|---------|---------|
| **后端开发工程师** | 日常调试、查日志、连数据库、操作 Docker | 工具太多，切换频繁 |
| **运维工程师 / SRE** | 管理多台服务器、批量操作、故障排查 | 缺乏统一管理面板，重复劳动多 |
| **全栈开发者** | 前后端开发 + 部署运维 | 终端/SSH/DB/Docker 各一个软件 |
| **技术团队 Leader** | 团队协作、知识沉淀 | 团队运维经验难以共享 |
| **独立开发者 / 创业团队** | 从开发、部署到线上排障一人包办 | 预算有限，需要一个软件覆盖完整链路 |
| **私有化客户 / 内网团队** | 内网服务器、数据库、容器和 AI 私有部署 | SaaS 工具不可用，数据安全要求高 |

### 1.6 核心使用场景

| 场景 | 用户目标 | OmniPanel 体验 |
|------|---------|----------------|
| 新项目部署 | 连接服务器、上传配置、启动容器、检查日志 | 从 SSH 到 SFTP 到 Docker Compose 到日志面板在同一工作区完成 |
| 线上故障排查 | 查监控、看日志、连数据库、执行修复命令 | AI 汇总当前主机、容器、日志和错误输出，生成排查路径 |
| 数据问题定位 | 查询数据库、比对数据、修复异常记录 | SQL 编辑器、数据网格、AI 解释和终端脚本联动 |
| 多服务器维护 | 批量执行命令、分发文件、比较执行结果 | 选择服务器分组后统一执行，结果按机器聚合和对比 |
| 团队经验沉淀 | 复用常见命令、排障流程和部署步骤 | 工作流模板、命令片段、会话记录和 AI 对话可沉淀为知识 |

### 1.7 产品形态定位

OmniPanel 应同时具备三种形态：

| 形态 | 定位 | 关键能力 |
|------|------|---------|
| 本地桌面工具 | 开箱即用的个人运维工作站 | 终端、SSH、数据库、Docker、监控、AI |
| 团队协作空间 | 小团队共享连接、脚本、工作流和知识 | 分组、权限、共享模板、审计记录 |
| 私有化运维入口 | 企业内网的统一工程入口 | 私有模型、离线部署、统一认证、策略管控 |

### 1.8 竞品分析

| 功能域 | 现有竞品 | OmniPanel 差异化 |
|--------|---------|-----------------|
| 终端 | Warp, iTerm2, Windows Terminal, Alacritty | AI 原生集成 + Blocks + 数据库/SSH 无缝切换 |
| SSH | Xshell, WindTerm, MobaXterm, Termius | 统一管理面板 + AI 辅助排错 |
| 数据库 | Navicat, DBeaver, DataGrip, Beekeeper | 终端+DB 一体化 + NL2SQL |
| Docker | Docker Desktop, Portainer, Lazydocker | 内嵌终端操作 + 可视化编排 |
| 服务器管理 | 宝塔, 1Panel, Cockpit | 客户端侧轻量管理 + 批量操作 |
| AI 助手 | Warp AI, GitHub Copilot CLI, Fig | 上下文感知的全栈运维 AI |
| All-in-One 工作台 | HexHub | 保持轻量精品定位，在上下文联动、AI 安全、终端体验和本地优先上做更深 |

### 1.8.1 HexHub 重点分析

HexHub 是目前与 OmniPanel 目标较接近的产品形态：它把 SSH、SFTP、数据库、Docker、AI Agent 放在同一个桌面客户端里，强调“不在 Xshell、Navicat、Docker Desktop、浏览器代理配置之间来回跳”的连续上下文体验。它验证了 All-in-One 开发运维工作台的真实需求，也说明市场上已经有人在收敛同类场景。

#### 产品结构

| 模块 | HexHub 表现 | 对 OmniPanel 的启发 |
|------|-------------|--------------------|
| SSH / SFTP 工作区 | 终端、远程文件、隧道、跳板机、lrzsz、X11、命令广播、多视图 | SSH 不能只做连接器，要成为远程开发运维的主入口 |
| 数据库客户端 | SQL 编辑器、结果集、表数据编辑、导入导出、数据同步、结构同步、跨库拷表 | 数据库模块要覆盖日常完整链路，尤其重视导入导出和同步性能 |
| Docker 管理 | 容器管理、Docker 文件管理、Compose 编辑 | Docker 与 SSH 主机、文件和日志需要自然联动 |
| AI Agent | 识别 SSH、Docker、远程文件、SQL 等动作，拆解成操作链并回显过程 | AI 应从问答升级为可审计的操作编排器 |
| 资产管理 | 分类目录、配置导入导出、资产同步、私有仓库同步 | 资源中心和工作区是 All-in-One 产品的骨架 |
| 多窗口 / 多视图 | 面向重度用户提供多视图和并行操作 | OmniPanel 需要兼顾轻量入口与专家级多任务 |

#### 值得汲取的精华

| 精华 | 说明 | OmniPanel 落地方式 |
|------|------|-------------------|
| 上下文不断掉 | SSH、Docker、数据库、文件和隧道在同一任务中连续发生 | 强化工作区模型和模块联动，减少复制粘贴 |
| 远程能力做完整 | SFTP、隧道、跳板机、命令广播、服务器状态面板都是高频刚需 | SSH 模块优先级上调，作为 Phase 1/2 的核心竞争力 |
| 数据库不是附属品 | 数据同步、结构同步、跨库迁移是实际付费价值点 | 数据库模块增加同步、迁移、导入导出性能目标 |
| AI 操作链 | AI 能拆解动作并展示执行过程，而不是只生成一段文本 | AI 模块增加计划、确认、执行、回放四段式交互 |
| 资产同步 | 多设备和团队场景需要资产可迁移、可同步 | 支持本地导入导出，后续提供私有仓库同步 |
| 专业场景优先 | 面向真正经常连服务器、查库、看容器的人 | OmniPanel 继续坚持小而全而优而美，避免泛化成低频功能集合 |

#### OmniPanel 需要形成的差异

| 方向 | 差异化要求 |
|------|------------|
| 更轻 | 控制包体、启动速度、默认界面复杂度，让新用户 2 分钟内完成首次连接 |
| 更优 | 把终端渲染、键盘操作、命令面板、错误反馈做到专业工具级别 |
| 更安全 | 生产环境保护、AI 操作分级、敏感信息脱敏、审计记录默认纳入设计 |
| 更美 | 统一视觉语言、密度配置、状态层级和高质量暗色主题 |
| 更工程化 | 工作区、资源中心、操作链、工作流、知识沉淀形成长期架构，而不是功能堆叠 |

### 1.9 核心差异化

| 差异化 | 说明 |
|--------|------|
| 从工具集合到工作流系统 | 不只是同时提供终端、SSH、DB、Docker，而是让它们围绕同一个任务上下文协作 |
| AI 不是聊天窗口 | AI 能读取当前终端输出、连接目标、数据库结构、容器状态和日志片段，给出可执行建议 |
| 本地安全基线 | 默认本地存储、系统 Keychain、敏感信息脱敏、危险命令确认，适合生产环境使用 |
| 跨平台一致体验 | Windows、macOS、Linux 使用统一操作模型和视觉语言 |
| 为小团队设计 | 既保留个人效率工具的轻量，又逐步支持团队共享、审计和规范化运维 |
| 可选协同层 | 个人用户本地完整可用，团队和企业按需启用 Hub、Web、手机端协同 |

## 2. 技术架构

### 2.1 技术选型

| 层 | 技术 | 说明 |
|----|------|------|
| 终端核心 | alacritty_terminal (Rust crate) | 生产级 VT 解析引擎，Apache 2.0 许可 |
| GPU 渲染 | wgpu | 跨平台 GPU 抽象层（Vulkan/Metal/DX12） |
| GUI 框架 | egui | 即时模式 GUI，代码量少，适合工具类 UI |
| 窗口管理 | winit | 跨平台窗口创建与事件循环 |
| SSH | ssh2-rs | Rust 原生 SSH2 协议实现 |
| 数据库驱动 | sqlx + 各数据库驱动 | 异步、编译时检查 SQL |
| Docker | bollard | Rust Docker Engine API 客户端 |
| 面板集成 | reqwest + 适配器模式 | 对接宝塔、1Panel 等服务器面板开放 API |
| AI 接口 | reqwest + CLI Agent Adapter + 自定义抽象层 | 支持云端 API、本地模型、本机 CLI Agent |
| 持久化 | rusqlite (本地配置) + 系统 Keychain (凭据) | 安全加密存储 |
| 语言 | Rust | 单二进制、零运行时依赖、内存安全 |

### 2.3 可选协同架构

OmniPanel 的默认形态是本地桌面客户端，所有核心能力在无服务端情况下也应可用。手机端作为个人用户也可使用的 Companion 入口，优先通过局域网与桌面端联动；云中继作为可选付费服务，只负责设备配对、消息转发和通知，不保存核心凭据。Hub、Web 端和执行代理作为 Team / Enterprise 阶段的可选协同层，避免过早引入完整服务端复杂度。

```
个人模式：
桌面客户端 ── 本地加密存储 / 系统 Keychain
     │
     ├─ Flutter 手机端 ── 局域网直连 / 二维码配对
     └─ 可选云中继 ── 消息转发 / 通知推送 / 在线状态

团队/企业模式：
桌面客户端 ─┬─ OmniPanel Hub ── Web 管理端
            ├─ 执行代理 Agent ── 内网服务器 / 数据库 / Docker
            └─ 手机端 ── 通知 / 审批 / 轻量处置
```

| 层级 | 职责 | 是否必需 |
|------|------|----------|
| 桌面客户端 | 完整终端、SSH、SFTP、数据库、Docker、AI 操作链 | 必需 |
| Flutter 手机端 | 监控、通知、审批、简易指令、工作流执行结果 | 可选 |
| 云中继 | 设备配对、消息转发、通知推送、在线状态 | 可选付费 |
| Hub 服务端 | 资产同步、权限、审计、工作流编排、通知、审批 | 可选 |
| 执行代理 Agent | 在内网执行定时任务、脚本和数据工作流 | 可选 |
| Web 端 | 团队管理、资产观察、审计、工作流管理 | 可选 |

### 3.0.1 工作区模型

工作区是 OmniPanel 的核心组织单位。一个工作区对应一个项目、一套环境或一个客户现场，内部包含相关连接、资源、历史和工作流。

| 对象 | 内容 |
|------|------|
| 连接 | SSH 主机、数据库连接、Docker Endpoint、服务器分组 |
| 资源 | 容器、镜像、数据表、文件路径、日志路径、常用命令 |
| 上下文 | 当前操作系统、Shell、环境标签、最近错误、最近查询、最近容器状态 |
| 工作流 | 部署流程、巡检脚本、排障步骤、数据修复模板 |
| 安全策略 | 生产环境确认规则、危险命令黑名单、脱敏规则、审计开关 |

### 3.0.2 模块联动

| 起点 | 联动动作 | 目标 |

### 3.7 AI 模块 (AI)

#### 3.7.1 模型支持

| Provider | 接入方式 | 说明 |

| OpenAI API | REST API | GPT 系列与 OpenAI 兼容接口 |

#### 3.7.2 AI Provider 类型

OmniPanel 的 AI 接入不应绑定单一云模型，而应抽象为统一 Provider。Provider 分为云端 API、本地模型和本地 CLI Agent 三类。

| 类型 | 代表 | 优点 | 风险与限制 |
|------|------|------|------------|
| 云端 API | OpenAI、Claude、DeepSeek、自定义 OpenAI 兼容接口 | 能力强、接入标准化、流式输出成熟 | 成本、网络、数据出站和密钥安全 |
| 本地模型 | Ollama、本地 OpenAI 兼容服务 | 数据不出本机，适合内网和敏感场景 | 模型能力和硬件性能受限 |
| 本地 CLI Agent | Cursor Agent、Claude Code、自定义 Agent CLI | 复用用户已有配置、账号、工具链和代码/终端上下文能力 | 进程权限、输出协议不稳定、不同工具能力差异大 |

#### 3.7.3 CLI Agent Adapter

CLI Agent Adapter 负责把外部命令行 AI 工具接入 OmniPanel。它不是直接接管这些工具，而是作为受控网关调用本机已安装、已授权的 Agent。

| 功能 | 优先级 | 描述 |

| Agent 检测 | P1 | 自动检测常见 CLI Agent 是否可用，例如 `cursor-agent`、Claude Code |

| 输入协议 | P1 | 支持 stdin、参数传入、临时文件、JSON payload 等输入方式 |
| 输出解析 | P1 | 支持纯文本、JSON、流式日志、退出码和错误输出解析 |
| 会话隔离 | P1 | 每次调用拥有独立工作目录、环境变量和取消控制 |
| 权限确认 | P0 | CLI Agent 执行命令、读取文件或修改资源前必须遵循 OmniPanel 的确认策略 |
| 审计记录 | P1 | 记录调用参数、上下文摘要、输出结果、用户确认和执行状态 |

#### 3.7.8 AI Agent 操作链

AI Agent 不应只回答“应该怎么做”，而要在用户授权下把跨模块任务拆成可审查、可执行、可回放的操作链。

| 阶段 | 描述 |
|------|------|
| 理解 | 读取用户目标和当前工作区上下文，例如 SSH 主机、容器、数据库、日志片段 |
| 计划 | 拆解为有顺序的操作集，标明每一步目标、风险和依赖 |
| 确认 | 高风险步骤需要用户逐项确认，可跳过、编辑或改为只生成脚本 |
| 执行 | 调用终端、SSH、Docker、SFTP、SQL 等模块执行具体动作 |
| 回显 | 实时展示每一步输入、输出、状态和失败原因 |
| 总结 | 执行完成后生成摘要、问题原因、后续建议，并可保存为工作流 |

#### 3.7.9 AI 安全与可信机制

| 机制 | 优先级 | 描述 |
|------|--------|------|
| 上下文最小化 | P0 | 只发送完成任务所需上下文，默认脱敏凭据和敏感字段 |
| 操作分级 | P0 | 将 AI 建议分为只读、低风险、高风险、不可自动执行 |
| 执行前预览 | P0 | 命令、SQL、Docker 操作在执行前展示影响范围 |
| 可解释建议 | P1 | AI 给出建议时说明依据，例如日志片段、指标异常、错误码 |
| 本地模型策略 | P1 | 对内网、生产、敏感数据场景优先提示使用本地模型 |
| 审计记录 | P1 | 保存 AI 生成内容、用户确认和执行结果，便于复盘 |
| CLI Agent 沙箱边界 | P1 | 外部 Agent 调用默认只读，写文件、执行命令、访问生产资源必须二次确认 |
| Provider 可见性 | P1 | 用户能看到当前请求发送给哪个 Provider，是云端、本地模型还是本地 CLI |
| 敏感环境变量过滤 | P1 | 调用 CLI Agent 时默认不透传全量环境变量，只传必要白名单 |

## 5. UI/UX 设计原则

### 5.1 设计理念

- **信息密度优先**：运维工具需要展示大量信息，不追求"留白"美学
- **键盘优先**：所有高频操作都应有键盘快捷键
- **一致性**：所有模块共享统一的布局、交互、主题系统
- **不打扰**：AI 建议是辅助而非强制，危险操作需确认
- **专业美学**：界面要像工程师的驾驶舱，克制、稳定、清晰、耐看
- **上下文可见**：用户始终知道自己在哪个工作区、哪台机器、哪个环境、哪个连接上操作
- **生产环境敬畏感**：生产环境必须在视觉上显著区分，让用户不容易误操作

### 5.2 整体布局

### 5.3 视觉语言

| 维度 | 要求 |
|------|------|
| 色彩 | 默认深色主题，强调色克制；生产环境使用稳定醒目的警示色 |
| 字体 | 终端使用高质量等宽字体，UI 使用清晰的系统字体 |
| 密度 | 支持紧凑、标准、舒适三档密度，满足不同屏幕和使用习惯 |
| 图标 | 图标风格统一，优先表达状态和对象类型，不做装饰性堆叠 |
| 动效 | 动效只用于状态变化和空间关系表达，避免影响专业工具效率 |
| 空状态 | 空状态提供下一步动作，例如新建连接、导入配置、打开工作区 |
| 状态反馈 | 连接中、执行中、失败、告警、只读、生产环境等状态必须一眼可见 |

### 5.4 交互原则

| 原则 | 描述 |
|------|------|
| 少搬运 | 用户不应在模块之间复制主机名、端口、日志、SQL 或错误信息 |
| 可预览 | 高风险操作执行前必须看到目标、命令、影响范围和回滚建议 |
| 可撤销 | 能撤销的操作提供撤销；不能撤销的操作提供确认和审计 |
| 可复用 | 高频操作应能保存为命令、工作流、连接模板或知识片段 |
| 快慢皆宜 | 鼠标用户能直观完成，新手不迷路；键盘用户能高速操作 |
| 不抢控制权 | AI 生成建议后等待用户确认，不主动执行破坏性动作 |

## 7. 商业化策略

### 7.1 定价模型

| 版本 | 价格 | 功能 |
|------|------|------|
| **Community** | 免费 | 终端 + SSH + 基础数据库 + 基础 Docker + 本地工作区 + 局域网 Companion |
| **Pro** | $12/月 或 $99/年 | 无限连接 + AI 功能 + 批量操作 + 高级 SSH + 工作流 + 云中继 |
| **Team** | $25/人/月 | Pro 全部 + 团队共享工作区 + 快捷脚本 + 数据工作流 + 审计 + SSO |
| **Enterprise** | 定制 | Team 全部 + Hub 私有化 + 执行代理 + 私有 AI + 定制开发 + SLA |

### 7.2 商业化原则

| 原则 | 说明 |
|------|------|
| 免费版可长期使用 | Community 必须是可信赖的基础工具，不能做成残缺试用版 |
| Pro 为效率付费 | 个人开发者为 AI、高级连接、批量操作、工作流、云中继等效率能力付费 |
| Team 为协作付费 | 团队为共享脚本、数据工作流、权限、审计、规范和知识沉淀付费 |
| Enterprise 为安全付费 | 企业为 Hub 私有化、执行代理、策略控制、合规、SLA 和定制集成付费 |
| 本地优先不变 | 商业化不能破坏本地优先和数据安全的产品信任 |

### 7.3 商业化安全边界

| 允许 | 不允许 |

| 学习 Warp 产品理念和功能设计 | 复制 Warp 源码 |
| 使用 alacritty_terminal 等开源 crate | 照搬 Warp 私有实现 |
| 实现类似功能（功能不受版权保护） | 抄袭 Warp UI 设计稿/样式 |
| 参考 Ghostty/WezTerm/Alacritty 等开源项目 | 大段引用后"洗"代码 |

## 8. 风险评估

| 风险 | 影响 | 概率 | 缓解策略 |
|------|------|------|---------|
| 功能边界失控，轻量体验被破坏 | 高 | 高 | 严格 MVP，终端+SSH 先行，高频场景优先，低价值功能延后 |
| 跨平台兼容性问题 | 中 | 中 | CI 多平台自动化测试，抽象层隔离平台差异 |
| 安全漏洞（凭据泄露） | 高 | 低 | 使用系统 Keychain，第三方安全审计 |
| 终端渲染性能不达标 | 高 | 低 | 复用 alacritty_terminal 核心，GPU 加速 |
| AI API 成本过高 | 中 | 中 | 支持本地模型，云端按需调用，缓存常见查询 |
| 竞品快速迭代 | 中 | 中 | 聚焦差异化（All-in-One + AI Native），快速响应 |
| Rust GUI 生态限制 | 中 | 中 | egui 成熟度足够，必要时局部用 WebView 补充 |
| 用户迁移成本 | 中 | 低 | 兼容导入主流工具配置，提供迁移向导 |
| All-in-One 体验割裂 | 高 | 中 | 以工作区、全局搜索、资源中心和模块联动作为产品主线 |
| UI 信息过载 | 中 | 中 | 提供密度配置、渐进披露、命令面板和清晰状态层级 |
| AI 误操作造成事故 | 高 | 中 | 操作分级、执行前预览、生产环境保护和审计记录 |
| 企业客户安全顾虑 | 高 | 中 | 本地优先、私有模型、禁用云端策略和敏感信息脱敏 |
| 第三方面板 API 变更 | 中 | 中 | 采用适配器模式，能力探测，版本兼容矩阵和失败降级 |
| 面板 Token 权限过大 | 高 | 中 | Keychain 加密、最小权限提示、操作审计和高风险二次确认 |
| 工作流误执行影响生产 | 高 | 中 | 环境标签、影响范围预览、审批流、回滚脚本和执行审计 |
| 过早服务端化拖慢产品 | 高 | 中 | 桌面端先闭环，Hub 放在 Future 阶段，个人模式不依赖服务端 |
| 多端权限模型复杂 | 中 | 中 | 先做最小角色模型，逐步扩展团队、企业和审计策略 |
| 功能丰富导致入口分散 | 中 | 中 | 通过命令面板、收藏夹、最近现场、草稿箱和时间线统一入口 |
| 云中继被误解为托管凭据 | 高 | 中 | 明确“云只做信使”，不保存核心凭据，所有敏感操作由桌面端或 Hub 执行 |
| 手机端误触发高风险操作 | 高 | 中 | 默认只读，低风险白名单，高风险必须二次确认或审批 |

复用原则：优先使用许可证友好、维护活跃、与 Rust 技术栈契合的库；对 GPL/AGPL 或许可不清晰项目，只做产品思路和交互流程参考，不直接复制代码；对安全、凭据、数据库写入、生产环境操作相关能力，必须二次封装并加入审计和确认机制。

#### 10.1.1 复用分级

| 级别 | 含义 | 处理方式 |
|------|------|----------|
| 直接使用 | 许可证友好、维护活跃、API 稳定 | 纳入技术选型，封装为内部接口 |
| 评估后使用 | 方向合适但成熟度、许可或维护状态需确认 | 做 PoC，确认性能、许可和跨平台表现 |
| 借鉴思路 | 产品形态、交互、架构值得学习，但不适合直接依赖 | 只参考思想，不复制源码、文案和 UI |
| 禁止复制 | 私有实现、GPL 风险不明确或商业许可不清 | 不复制代码，不照搬 UI，必要时只做竞品分析 |
